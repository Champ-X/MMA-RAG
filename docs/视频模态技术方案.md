## 一、 核心设计理念

本方案的核心思想是**“物理不切，语义切分，按帧锚定”**。

利用多模态大模型（如 qwen3.5-plus 等）作为“分析大脑”，直接理解视频并输出带有精确时间戳的结构化场景（Scene）与关键帧（Keyframe）。底层对象存储仅保留一份完整视频流，向量数据库则以“关键帧”为最小粒度进行存储，实现“全局语义与局部视觉”的完美统一。

## 二、 存储底座规划 (MinIO)

视频文件本身极大，我们采用解耦存储策略：

- `**videos/` (原始视频):** 仅上传并存储一份完整的原始视频文件（如 `peaky_blinders.mp4`）。不进行物理切块落盘，节省海量空间。与现有项目一致：对象名 `videos/{file_id}_{原始文件名}`。
- **关键帧图片存放路径（与现有 `images/` 区分）：** 根据 MLLM 给出的时间戳，后台提取对应时间点的画面（如 `.jpg`），建议使用**独立子路径**，避免与知识库内「独立上传的图片」或「文档内嵌图片」混用。推荐两种之一：
  - **方案 A：** `videos/{file_id}/keyframes/{segment_id}_{timestamp}.jpg`（关键帧与视频同前缀，便于按 file_id 清理）。
  - **方案 B：** `images/video_keyframes/{file_id}/{segment_id}_{timestamp}.jpg`（仍在 images 桶下但用固定前缀区分）。
- 上述路径写入 Qdrant payload 的 `frame_image_path`，供检索结果展示与「一键跃迁」播放。

## 三、 数据摄入与解析流水线 (Ingestion & Parsing)

这是整个方案的工程难点与核心所在，负责将长视频转化为结构化数据。

### 3.1 长短视频策略与长视频滑动窗口 (Sliding Window)

- **长短视频分流（与现有实现衔接）：** 建议按视频时长选择策略。例如：时长 ≤ 某阈值（如 2 分钟）可沿用现有「固定间隔抽帧 + 每帧 VLM 描述 + 整体描述 + 一视频一点」流水线，以降低时延与成本；超过阈值则走本方案的「滑动窗口 + MLLM 语义解析 + 一关键帧一点」。阈值可在配置中设定。
- **长视频滑动窗口：** 为了突破 MLLM 的上下文长度限制和防止 OOM（显存溢出），采用内存级虚拟切片处理长视频（如电影）。
  1. **定义窗口：** 设定固定处理窗口（如 480 秒/8分钟）与重叠区（如 10 秒）。
  2. **内存抽流：** 使用 `ffmpeg` 将原始视频流切分为虚拟的 Chunk (例如：Chunk 0: 0-480s, Chunk 1: 470-950s)，分批送入 MLLM。

### 3.2 MLLM 语义结构化解析 (Semantic Parsing)

通过强约束 Prompt 与 Structured Output（如 JSON Mode），强制 MLLM 输出规范的解析结果。

- **Prompt 核心要求：** “按时间戳顺序输出 JSON 数组。划分逻辑场景，并根据画面丰富度在场景内提取 1-4 个关键帧。”
- **输出结构示例 (局部时间)：**
  ```json
  [
    {
      "start_time": 0.0,
      "end_time": 15.0,
      "scene_summary": "汤米和女子在废墟中乘车，随后爆发打斗...",
      "keyframes": [
        { "timestamp": 2.0, "description": "女子穿过瓦砾街道，..." },
        { "timestamp": 10.0, "description": "汤米被打倒在地，..." }
      ]
    }
  ]
  ```

#### 3.2.1 MLLM Prompt 设计（强约束版）

以下为可直接用于调用的 Prompt 设计，便于与 JSON Mode / Structured Output 配合，得到稳定可解析结果。

---

**【System / 角色设定】**

```
你是一名专业的视频内容分析师。你的任务仅有一个：观看输入的视频片段后，输出**唯一**一段合法的 JSON，且不包含任何 JSON 之外的文字、解释或 Markdown 标记。

规则概览：
- 将本片段按「逻辑场景」切分为若干连续区间，每个区间一个场景。
- 每个场景内根据「画面信息量」与「叙事代表性」选取 1～4 个关键帧，并给出精确到秒（可为小数）的时间戳与一句画面描述。
- 所有时间戳均相对于**本段视频的 0 秒**。只输出一个 JSON 数组，不要用 ```json 包裹，不要输出其他任何内容。
```

---

**【User / 任务与格式说明】（与视频一起传入）**

```text
请对当前这条视频片段进行「场景划分 + 关键帧提取」，并严格按下列要求输出。

## 一、输入说明
- 本片段是长视频中的一段连续剪辑，时长约 {chunk_duration_seconds} 秒。
- 若上游告知本段在完整视频中的起始偏移，仅用于你理解上下文；你输出的时间戳仍以本段 0 秒为起点。

## 二、场景划分规则（满足其一即可考虑切分）
- **空间/地点变化**：明显换景、转场（如从室内到街道）。
- **时间或叙事跳跃**：闪回、切到另一条故事线、黑场/白场后重新开始。
- **主体或事件转折**：主要人物变化、核心动作/事件结束并开启新事件。
- **镜头语言突变**：景别或运镜风格明显改变且伴随语义转折（如从特写切到全景并换人）。
不要因短暂切镜或同一场景内的小幅运镜就切分；单场景至少覆盖数秒以上。

## 三、scene_summary 写作要求（重要：须尽可能详尽）
- **scene_summary 是该视频片段的场景级描述，用于后续向量检索与 LLM 上下文，必须尽可能详尽、信息充分。**
- 建议涵盖：**地点/环境**（室内/外、具体场所、时间氛围）、**出现的人物**（身份、穿着、数量、大致关系）、**主要动作与事件**（谁对谁做了什么、关键转折）、**情绪或氛围**（紧张、平静、冲突等）、若有明显**对白、字幕或旁白要点**可简要概括。
- 长度：以 2～6 句话或 80～200 字为参考，宁详勿略；避免仅用一句笼统概括导致检索时信息不足。
- 语言：中文、连贯段落、无换行、无多余引号；与 keyframe 的 description 区分——scene_summary 是「整段场景的完整叙述」，description 是「单帧画面的单句描述」。

## 四、关键帧选取规则（每场景 1～4 帧）
- **信息量优先**：选择能代表该场景「谁、在哪、在做什么」的瞬间，避免模糊、黑屏、重复构图。
- **时间分布**：在场景的 [start_time, end_time] 内尽量分散选取，避免多帧挤在同一秒。
- **描述要求**：每个 keyframe 的 description 为一句话、中文、客观描述画面内容（人物/物体/动作/布局/氛围），不推测剧情或心理，不写“可能”“似乎”。时间戳为浮点数秒，且必须落在该场景的 start_time 与 end_time 之间。

## 五、输出格式（必须严格遵守）
输出**仅**一个 JSON 数组，每个元素形如：
{
  "start_time": <浮点数，本场景起始秒数>,
  "end_time": <浮点数，本场景结束秒数>,
  "scene_summary": "<详尽场景描述：见上文「三、scene_summary 写作要求」，2～6 句话覆盖地点、人物、事件、氛围等，用于检索与上下文>",
  "keyframes": [
    { "timestamp": <浮点数>, "description": "<一句画面描述>" },
    ...
  ]
}

约束：
- 数组按 start_time 严格递增；场景之间时间连续、不重叠（相邻场景的 end_time 等于下一场景的 start_time，或允许极小缝隙）。
- **scene_summary 必须按第三节要求尽可能详尽**；description 为一句话、简洁。二者均使用中文，无换行、无多余引号。
- 每个场景的 keyframes 数量为 1～4，timestamp 严格落在 [start_time, end_time] 内。
- 不要输出任何 JSON 之外的字符（包括 ``` 或“以下是 JSON”等说明）。
```

**【边界与降级】**

- **整段无明显场景变化**：可只输出一个场景，start_time=0，end_time=本段时长，keyframes 仍按信息量选 1～4 帧。
- **极短片段（如 <30 秒）**：可单场景多关键帧，或 1～2 个场景，避免过度切分。
- **解析失败或非 JSON**：下游用正则或二次解析兜底；若检测到 ```json 包裹，先剥掉再解析。

### 3.3 全局时间轴对齐与边界缝合 (Temporal Alignment)

1. **全局映射：** 针对后续的每一个 Chunk，执行时间戳算术偏移。设当前 Chunk 在原视频的起始偏移量为 $T_{offset}$，则实际时间为：
  $$T_{global} = T_{local} + T_{offset}$$
2. **场景去重(可选)：** 遍历所有 Chunk 聚合后的 JSON。若发现相邻两个场景的 `[start_time, end_time]` 在重叠区内有交叉，且 `scene_summary` 语义相似度极高，则在代码层将它们融合为一个更长的完整场景，合并其关键帧。

## 四、 特征向量提取 (Feature Extraction)

拿到全局对齐的 JSON 结构后，对每一个**关键帧 (Keyframe)** 执行多维特征提取。为避免 **scene_summary 与 frame description 拼接**带来的信息重复和帧级描述被长文本淹没，采用**场景与帧分离向量化**：

1. **截取画面：** 根据全局 `timestamp`，利用 OpenCV 瞬间定位并提取该帧画面，转存 MinIO。
2. **视觉流 (CLIP Vector)：** 将该帧画面直接送入 CLIP 模型，提取得到 768 维的纯视觉特征向量 `clip_vec`。
3. **场景语义流 (scene_vec)：** 仅将**该关键帧所属场景**的 `scene_summary` 送入 Text Embedding 模型，得到 4096 维向量 `scene_vec`。同一场景下的多个关键帧共享**同一段** scene_summary，因此可复用同一 scene_vec，无需对同一场景重复编码。
4. **帧语义流 (frame_vec)：** 仅将该关键帧的 `description`（帧级描述）送入 Text Embedding 模型，得到 4096 维向量 `frame_vec`。保证帧级细节独立表达，不被场景摘要稀释。

这样检索时既可命中「场景级」查询（如“废墟里打斗那段”），也可命中「帧级」查询（如“汤米被打倒在地的画面”），且无重复与淹没问题。

## 五、 向量数据库架构设计 (Qdrant)

**粒度定义：1 个关键帧 = 1 个 Point。** (多个属于同一场景的 Point 共享相同的场景级 Payload 数据)。

### 与现有 video_vectors 的衔接

- 集合名仍为 `video_vectors`。本方案下每个 Point 为**三向量**：`scene_vec`(4096)、`frame_vec`(4096)、`clip_vec`(768)，需在现有 `text_vec`+`clip_vec` 基础上扩展 schema（新增 `scene_vec`、`frame_vec`，或以二者替代单一 `text_vec`）。检索需支持三路 Prefetch + RRF。
- 采用「一关键帧一点」后，需扩展 payload 并支持按关键帧批量写入；**删除/更新**时按 `file_id` 删除该视频下所有 Point，与现有按文件维度的删除逻辑一致。

### Qdrant Collection Schema 示例（scene / frame 分离向量）

```json
{
  "vectors": {
    "scene_vec": { "size": 4096, "distance": "Cosine" },
    "frame_vec": { "size": 4096, "distance": "Cosine" },
    "clip_vec": { "size": 768, "distance": "Cosine" }
  },
  "payload_schema": {
    "kb_id": { "data_type": "keyword", "index": true },
    "file_id": { "data_type": "keyword", "index": true },
    "segment_id": { "data_type": "keyword" },
    "scene_start_time": { "data_type": "float" },
    "scene_end_time": { "data_type": "float" },
    "scene_summary": { "data_type": "text", "index": "text" },
    "frame_timestamp": { "data_type": "float" },
    "frame_description": { "data_type": "text" },
    "frame_image_path": { "data_type": "text" },
    "duration": { "data_type": "float" },
    "video_format": { "data_type": "keyword" },
    "created_at": { "data_type": "text" }
  }
}
```

## 六、 检索与生成 (Retrieval & Generation)

当用户发起查询（如：“汤米被打倒在地的画面在哪？”）时：

1. **多路召回 (Hybrid Search)：**
  - 将 Query 转为**同一** Text Embedding 向量（与 scene_vec、frame_vec 同模型），分别检索 `scene_vec` 与 `frame_vec`。
  - 将 Query 转为 CLIP 文本向量检索 `clip_vec`。
  - 对 **scene_vec / frame_vec / clip_vec** 三路结果做 RRF 融合，得到 Top-K 个关键帧 Point。这样既支持“整段场景”类查询，也支持“某一帧画面”类查询，且帧级描述不会被场景摘要淹没。
2. **场景聚合去重 (Group By Segment)：**
  - 检查召回的 Top-K 结果。若发现多个 Point 的 `segment_id` 相同，说明它们属于同一个场景片段。
  - 将它们合并为一个 Context 块，保留得分最高的那一帧作为核心代表。
3. **LLM 最终生成与前端组装：**
   - **Context 给 LLM：** 从命中 Point 的 payload 中取出 `scene_summary`、`frame_description` 以及对应的时间戳，一并传入 LLM（scene_summary 提供场景级上下文，便于生成连贯回答）。
  - **输出与交互：** LLM 生成精准回答，并在 Markdown 中嵌入 `frame_image_path` 展示画面。前端播放器捕获 `frame_timestamp`，允许用户点击后“一键跃迁”至该秒数播放。
4. **与现有检索层的融合：** 视频检索结果与 Dense、Sparse、Visual、Audio 等一起进入 `HybridSearchEngine._fuse_results`，按 RRF 融合。建议与现有 `_video_search` 统一入口，并根据意图（若有 `video_intent`）在 limit 与 score_threshold 上做与图片/音频类似的显式/隐性区分，权重可与现有视频权重（如 1.1）保持一致或可配置。
