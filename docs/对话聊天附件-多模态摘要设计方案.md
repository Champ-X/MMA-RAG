# 对话聊天附件：多模态摘要设计方案

本文档描述**对话输入区**仅支持**图片**与**音频**两类附件时的产品与技术方案。附件**不参与分块、向量化与写入 Qdrant**；通过**多模态 / 语音理解模型**生成**长度可控的文本摘要**，分别注入**查询理解（One-Pass 意图与检索策略）与最终生成 Prompt**，避免将原始长文本暴力塞进上下文。

---

## 一、目标与边界

### 1.1 目标

- 用户在同一轮提问中可附带若干图片/音频，模型回答能**结合附件语义**与**知识库检索结果**。
- 附件侧**不建索引、不入库**，仅当次请求内使用（除非产品另行要求「会话级缓存」，见下文可选能力）。
- 摘要长度有**硬预算**，保证意图识别与最终生成总上下文可控。

### 1.2 非目标（明确不做）

- 不支持文档类附件（PDF/DOCX/PPTX/TXT 等）在对话中的上传与解析。
- 附件内容**不**进入 `text_chunks` / `image_vectors` / `audio_vectors` 等向量存储。
- 不要求附件与知识库结果做向量空间的「联合检索」；检索仍只针对已选知识库。

### 1.3 与现有系统的关系

- **知识库侧**多模态流水线（ingestion、Qdrant、HybridSearch）保持不变；本方案仅在 **chat 路径**增加「会话附件 → 摘要 → 两阶段注入」。
- 实现时可**复用**已有能力思路，但**调用链独立**，避免误触入库逻辑：
  - 图片：可参考 `IngestionService._generate_image_caption` 所使用的 `task_type="image_captioning"` 与多模态消息格式（见 `backend/app/modules/ingestion/service.py`）。
  - 音频：可参考 `_transcribe_audio` 及「转写 + 简短描述」的解析方式（同文件内音频处理），但 chat 附件只需**摘要级输出**，不必写 MinIO/Qdrant，不需要转写内容，有摘要信息jiu'zu。

## 二、产品约束


| 项     | 建议值                                                    | 说明                                                                             |
| ----- | ------------------------------------------------------ | ------------------------------------------------------------------------------ |
| 允许类型  | 图片、音频                                                  | MIME 白名单校验，拒绝其它类型                                                              |
| 图片格式  | `image/jpeg`, `image/png`, `image/webp`, `image/gif` 等 | 与前端 `accept`、后端校验一致                                                            |
| 音频格式  | `audio/flac`, `audio/wav`, `audio/mp3`等                | 以 ASR/音频 MLLM 实际支持为准"mp3", "wav", "m4a", "flac", "aac", "ogg", "wma", "opus", |
| 单文件大小 | 建议图片 ≤ 2MB，音频 ≤ 5MB（可配置）                               | 控制上传耗时与模型输入限制；具体数值可在实现阶段按网关/模型上限调整                                             |
| 数量上限  | 建议每轮 ≤ 3 个                                             | 与现前端「最多 3 个附件」习惯对齐；超限直接 400                                                    |
| 摘要总预算 | 建议**单附件**摘要 ≤ 300 汉字                                   |                                                                                |

## 三、总体流程

```
用户输入文本 + 0..N 个附件
        ↓
[1] 校验类型/大小/数量
        ↓
[2] 并行或串行：对每个附件调用「摘要管线」（仅内存/临时文件，不入库）
        ↓
[3] 合并为结构化「附件上下文」attachment_context（见第四节）
        ↓
[4] 查询理解：One-Pass 模板增加「附件摘要」区块 → 产出 refined_query、keywords、多模态意图等
        ↓
[5] 检索：逻辑与现网一致（仅依赖步骤 4 的检索策略与 kb）
        ↓
[6] 生成：在 ContextBuilder / 最终 messages 中追加「本轮用户附件摘要」区块，与检索材料并列
```

要点：

- 步骤 2 的耗时会体现在首包延迟上，可在 SSE 中增加阶段事件（如 `attachment_summary`，可选）。
- 步骤 4、6 使用的是**同一份** `attachment_context`（或生成阶段使用略长的版本需在文档中约定是否允许；默认**同一份**以省算力）。

## 四、附件摘要管线（核心）

### 4.1 输出结构（建议）

每条附件生成一条记录，供下游统一拼接：

```json
{
  "index": 1,
  "modality": "image | audio",
  "filename": "string",
  "summary": "string",
  "meta": {
    "duration_sec": null,
    "language_hint": null
  }
}
```

- **图片**：`summary` 为 VLM 生成的**短摘要**（见 4.2），强调与用户提问、知识检索相关的语义，而非 ingestion 用的超长「检索导向」caption（若复用同一 prompt，需在 chat 场景换用**短摘要模板**或后处理截断）。
- **音频**：`summary` 建议为「内容概述 + 关键信息」（说话人角色、主题、专有名词、数字等，**不要把完整 ASR 原文**直接进 One-Pass。

### 4.2 图片摘要

- **模型**：使用配置中的视觉/多模态对话能力（与 `image_captioning` 任务对齐或单独配置 `chat_attachment_image`）。
- **Prompt 原则**：
  - 明确字数上限（例如「200 字以内」）。
  - 要求：客观描述画面主体、文字（OCR 要点）、图表类型与结论；禁止套话开头。
  - 可选：传入用户**当前输入文本**一句，提示「摘要需便于结合用户问题做检索与回答」（注意勿超长）。

### 4.3 音频摘要

- **优先**：支持音频输入的多模态模型**一步**输出「有限长度摘要」（+ 可选短转写要点列表）。不要转写文本只要**音频描述**。

### 4.4 失败与降级

- 单附件失败：该条 `summary` 记为「（摘要失败：原因简述）」或置空，**不阻断**整轮对话（可配置为严格模式则 4xx）。
- 全部失败：等价于无附件，仅用户文本走检索与生成，并可在 UI 提示。

## 五、查询理解阶段（One-Pass）注入

### 5.1 输入扩展

在现有 `ONE_PASS_INTENT_TEMPLATE`（`backend/app/core/llm/prompt.py`）的输入部分增加独立区块，例如：

```text
## 用户本轮附带的图片/音频摘要（已由系统生成，非用户原文）
{attachment_context_block}
```

`attachment_context_block` 为由第四节结构渲染的纯文本，例如：

```text
[附件1 图片 示意图.png] …摘要…
[附件2 音频 会议录音.m4a] …摘要…
```

### 5.2 期望行为

- **指代消解**：用户说「这张图里写的参数是什么？」时，`dense_query` / `sparse_keywords` 应融合摘要中的实体与数值。
- **多模态意图**：`visual_intent` / `audio_intent` 仍针对「知识库检索是否要强查图/音」；**用户已上传图片**时，可在 reasoning 中说明「用户已提供现场图，回答应优先依据附件摘要」，但**不必**因此强制 `explicit_demand`（避免无意义拉高 KB 图检权重），具体规则可在实现时微调并写入模板说明。

### 5.3 接口层参数

- `RetrievalService.search` / `search_stream` 的 `_preprocess_query` 增加可选参数，例如 `attachment_context: Optional[str]`。
- 无附件时传空，行为与现网完全一致。

## 六、生成阶段注入

### 6.1 位置与格式

- 在 `ContextBuilder` 或等价模块中，在「用户问题」与「检索到的知识库材料」之间（或材料之后、用户问题复述之前）插入固定标题区块，例如：

```text
【本轮用户上传的媒体摘要（非知识库检索结果，请勿标注为知识库引用编号）】
...
```

- 系统说明中要求模型：**回答须结合该区块与检索材料**；引用编号**仅**对应知识库检索片段，附件内容用自然语言指代（「根据您上传的图片…」）。

### 6.2 与流式生成一致

- `GenerationService.stream_generate_response` 的上下文构建需接收与同步接口相同的 `attachment_context`（或结构化列表 + 统一渲染）。

## 七、API 与前端

### 7.1 传输方式

- **推荐**：`POST /chat/stream`（或统一 chat 入口）使用 **multipart/form-data**：
  - 字段：`message`, `sessionId`, `knowledgeBaseIds`, `model`（可选）
  - 文件：`files` 多个二进制流
- **不推荐**：GET + Query 传 message（现有 SSE）无法携带文件；若保留 GET，需另设「先 POST 附件拿 `attachment_token`」的两段式，复杂度高。

> 实现迁移时：前端 `EventSource` 无法直接发 multipart，需改为 **fetch + ReadableStream 解析 SSE** 或 **分步 API**（先摘要，再带 `attachment_digest` 拉流）。

### 7.2 前端改造要点

- `accept` 仅图片、音频；选择后立即校验大小与类型。
- 在内存保留 `File` 对象直至发送成功（当前若只存 `name/size` 则无法满足上传）。
- 发送成功后是否清空附件：与产品一致；会话历史是否展示「本轮附件名」可仅存元数据，**不必**把摘要存前端长期缓存（摘要已在服务端进入当轮 Prompt）。

---

## 八、安全与合规

- 文件类型以 **MIME + 魔数**双校验，防止扩展名伪造。
- 上传大小在网关与应用层同时限制；超时单独配置（音频可能较长）。
- 日志中不打印完整摘要外的原始二进制；摘要可打截断日志。

---

## 九、可选扩展（非首期必做）

- **会话级缓存**：同一附件 hash 在短 TTL 内复用摘要，减少重复调用（需 Redis 或进程内 LRU，注意隐私）。
- **用户显式开关**：「附件仅用于理解问题，不要让回答引用附件」之类（通过模板开关控制生成段是否注入）。

## 十、实现任务拆分（供排期）


| 序号  | 项                                       | 说明                                    |

## 十一、验收要点

- 仅图片/音频可上传，其它类型拒绝。
- 单附件超大、超多附件有明确错误提示。
- 有附件时，One-Pass 产出的 `dense_query` / 关键词能体现附件中的实体或数据（可用固定用例回归）。
- 生成回答能使用附件摘要信息，且知识库引用编号不与附件块混用。
- 确认附件处理**无** Qdrant/知识库向量写入副作用（监控或集成测试断言）。
