# 知识库画像路由优化分析

## 一、问题背景

当前基于知识库画像的在线路由在测试中表现出以下问题：

1. **单库错选**：Qwen/阿里巴巴相关查询被路由到 DeepSeek 知识库（kb1）
2. **多库漏选**：应同时路由到 DeepSeek+Qwen 的查询只路由到 DeepSeek

本文档分析根因并提出优化方案，供后续实现参考。

---

## 二、失败模式概览

| 类型 | 示例 | 现象 |
|------|------|------|
| **单库错选** | 「阿里巴巴人工智能技术的进化史」「Qwen3-VL的架构图」 | 应路由到 kb2 (Qwen)，实际路由到 kb1 (DeepSeek) |
| **多库漏选** | 「DeepSeek和Qwen都有视觉模型」 | 应路由到 kb1+kb2，实际只路由到 kb1 |

---

## 三、当前实现简述

### 3.1 画像生成（portraits.py）

| 环节 | 逻辑 |
|------|------|
| 采样 | N < 5000 全量；否则蓄水池采样约 20%，min 50、max 1000 |
| 聚类 | K-Means，K = √(N/2)，限制在 [2, max_kb_portrait_size] |
| 主题摘要 | 每簇取距中心最近的 5–10 个 chunk，LLM 生成 topic_summary |
| 存储 | 摘要向量化后存入 kb_portraits |

### 3.2 路由打分（router.py）

| 环节 | 逻辑 |
|------|------|
| 检索 | 查询向量化 → kb_portraits 全表 TopN=30 相似检索 |
| 打分 | Score(KB) = Σ (Similarity × log(ClusterSize+1))，按 kb_id 求和 |
| 归一化 | Min-max 到 [0, 1] |
| 决策 | max_raw < 0.08 → 全库；gap ≥ 0.25 → 单库 dominant；否则 top2 多库 |

---

## 四、根因分析

### 4.1 大库垄断

- kb1 (DeepSeek) 画像数量或 cluster_size 普遍更大
- `log(cluster_size+1)` 使大簇在聚合得分中占优
- kb1 在 Top30 中节点更多，求和后总分更高

### 4.2 主题重叠

- DeepSeek 与 Qwen 均为大模型主题，embedding 空间接近
- 画像摘要可能偏向泛化描述（如「AI 模型」），区分度不足

### 4.3 多库决策过严

- `gap ≥ 0.25` 即判为 single_kb_dominant
- 第二名得分虽不低（如 0.29），但因 gap 大被排除
- 缺乏「第二名足够高则允许多库」的规则

---

## 五、优化方向与方案

### 5.1 画像侧：缓解大库垄断

| 方案 | 说明 |
|------|------|
| **A1. 聚类数按 KB 均衡** | 对每个 KB 限制最大聚类数（如 `max_per_kb`），避免大 KB 画像过多 |
| **A2. 弱化 cluster_size 权重** | 将 `log(cluster_size+1)` 改为更平缓形式，如 `log(1 + log(cluster_size+1))` 或固定权重 |
| **A3. TopN 内按 KB 截断** | 每个 KB 在 TopN 中最多保留 M 个节点（如 M=10），防止单 KB 占满 Top30 |
| **A4. 提高画像粒度** | 对相似主题 KB 适当增加聚类数，使画像更细分、区分度更高 |

### 5.2 路由侧：打分与决策优化

#### 5.2.1 多库识别不足

| 方案 | 说明 |
|------|------|
| **B1. 多库阈值放宽** | 当第二名归一化得分 > 某阈值（如 0.2/0.25）时，即使 gap ≥ 0.25 也允许多库，或新增 `ROUTING_MULTI_KB_MIN_SCORE` |
| **B2. 绝对得分阈值** | 在归一化前：若有多于 1 个 KB 的 raw 得分 > 某阈值，则视为多库候选 |
| **B3. 关键词辅助** | 检测「和」「都」「两者」等，倾向多库；或对「DeepSeek 和 Qwen」类查询强制多 KB |

#### 5.2.2 相似主题区分

| 方案 | 说明 |
|------|------|
| **B4. Top1 节点信号** | 若 Top1 节点属于 kb2，即使 kb1 总分更高，也可考虑 kb2 或 dual，作为「最相关单库」的补充 |
| **B5. 相似度加权** | 用 `sim²` 或 `sim × log(1+sim)` 提高高相似节点的权重，降低 cluster_size 影响 |
| **B6. 动态 TopN 截断** | 按首次出现明显得分断档的位置截断，减少弱相关节点对聚合的干扰 |

### 5.3 参数可配置化

| 参数 | 当前值 | 建议 |
|------|--------|------|
| `ROUTING_TOP_N` | 30 | 可配置，建议 20–50 |
| `ROUTING_GAP_DOMINANT` | 0.25 | 可配置，建议 0.15–0.35 |
| `ROUTING_ALL_LOW_THRESHOLD` | 0.08 | 可按实际分布调优 |
| **新增** | - | `ROUTING_MULTI_KB_MIN_SCORE`：第二名归一化得分下限，超过则允许多库 |
| **新增** | - | `ROUTING_MAX_NODES_PER_KB`：每个 KB 在 TopN 中最多保留的节点数 |

### 5.4 画像生成侧

| 方案 | 说明 |
|------|------|
| **C1. 增加每簇样本数** | 将 NEAREST_PER_CLUSTER 从 5–10 提高到 8–15，使 LLM 摘要更充分 |
| **C2. 优化 prompt** | 明确要求突出「与其他主题的差异」，强调品牌/产品名（如 Qwen、DeepSeek、阿里） |
| **C3. 主题去重** | 生成画像后对 topic_summary 做相似度检查，合并或删减高度重复的簇 |
| **C4. 分层画像** | 一级粗粒度主题（如「大模型」），二级细粒度（如「Qwen 系列」「DeepSeek 系列」），路由时可先粗后精 |

### 5.5 测试与数据

| 方案 | 说明 |
|------|------|
| **D1. 放宽断言** | 对模糊查询（如「阿里巴巴人工智能技术的进化史」可能涉及 Qwen），接受 kb1 或 kb2 |
| **D2. 调整期望** | 将「DeepSeek和Qwen都有视觉模型」的期望改为 `[kb1]` 或 `[kb1, kb2]` 均可 |
| **D3. 回归数据集** | 用真实 query 日志建立回归集，区分「必须正确」「可多选」「可降级」等档位 |

## 六、优化优先级建议

| 优先级 | 方向 | 预期收益 |
|--------|------|----------|
| **P0** | B1：多库阈值放宽，新增 `ROUTING_MULTI_KB_MIN_SCORE` | 显著改善应选多库却只选单库的问题 |
| **P0** | A3：TopN 内按 KB 截断（`ROUTING_MAX_NODES_PER_KB`） | 缓解大 KB 垄断，kb2 有机会参与竞争 |
| **P1** | A2：弱化 cluster_size 权重 | 减轻大库天然高分 |
| **P1** | C2：优化画像生成 prompt | 提升主题区分度，减少跨 KB 混淆 |
| **P2** | B4：Top1 节点信号 | 增强对「最相关单库」的识别 |
| **P2** | 参数配置化 | 便于调参与 A/B 测试 |

---

## 七、总结

当前问题的核心在于：

1. **画像与打分**：大 KB 画像更多、cluster_size 更大，在 Top30 和聚合得分上占优
2. **路由决策**：gap 阈值偏严，缺少「第二名足够高则多库」的规则

建议优先实现 **B1（多库阈值放宽）** 和 **A3（TopN 内按 KB 截断）**，再视效果逐步调优 cluster_size 权重、画像 prompt 及参数配置化。
