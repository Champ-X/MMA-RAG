# 知识库智能路由：设计方案与实现细节

本文档描述当前代码中 **知识库智能路由** 的端到端设计、算法参数与调用关系，实现入口为 `backend/app/modules/knowledge/router.py` 中的 `KnowledgeRouter`。

> **与旧文档的关系**：`ROUTING_STRATEGY_ANALYSIS.md` 部分段落描述的是历史实现与策略的对比；`KB_PORTRAIT_ROUTING_OPTIMIZATION.md` 中「路由打分」小节仍写旧公式（按 `cluster_size` 求和）。**以本文与 `router.py` 源码为准**。

---

## 1. 设计目标

- **自动选库**：在未显式指定知识库时，根据用户查询语义，从多个知识库中选出 1～2 个最相关库（或退化为全库），再交给混合检索。
- **缓解大库垄断**：画像很多的知识库若在 TopN 中命中节点过多，简单「按库求和」会系统性抬高该库分数；当前实现通过 **每库仅取前 K 个相似节点 + 位置衰减加权平均** 抑制该问题。
- **可解释与可观测**：通过 `routing_method`、归一化分、`audit_log` 与结构化日志便于排查。
- **与检索链路一致**：路由使用与文档 chunk 相同的 embedding 管线；检索阶段将业务侧 `kb_id` 解析为 Qdrant 中实际存储的 `kb_id`。

## 2. 整体架构与数据基础

### 2.1 画像集合 `kb_portraits`

- 由 `PortraitGenerator`（`backend/app/modules/knowledge/portraits.py`）对单个知识库采样向量、K-Means 聚类、LLM 生成主题摘要后，将摘要向量写入 Qdrant 集合 **`kb_portraits`**。
- 每条画像点的 payload 至少包含 **`kb_id`**、**`cluster_size`** 等；路由的 **全局相似检索** 主要使用 **向量相似度**；当前打分公式 **不再** 将 `cluster_size` 乘入得分（见第 4 节）。

### 2.2 在系统中的位置

```mermaid
flowchart LR
  subgraph preprocess [检索预处理]
    Q[用户查询]
    IR[IntentProcessor]
    RW[QueryRewriter]
    RQ[refined_query]
    Q --> IR --> RW --> RQ
  end
  subgraph route [智能路由]
    E[embed refined_query]
    T[search_kb_portraits_topn]
    AGG[按 kb 聚合得分]
    DEC[归一化 + 差距决策]
    RQ --> E --> T --> AGG --> DEC
  end
  subgraph retrieve [混合检索]
    RES[resolve_to_qdrant_kb_ids]
    HS[HybridSearchEngine.search]
    DEC --> RES --> HS
  end
```

- **入口**：`RetrievalService.search` / 流式路径在预处理完成后调用 `_route_to_knowledge_bases(preprocessing_result["refined_query"], kb_context)`（见 `backend/app/modules/retrieval/service.py`）。
- **含义**：路由使用的文本是 **改写后的 `refined_query`**，与 dense 检索主查询对齐，而非原始用户原句（除非改写失败回退）。

## 3. 路由模式概览

| 条件 | 行为 | `routing_method` |
|------|------|------------------|
| `kb_context` 含非空 `kb_ids` | 直接使用这些库，置信度均为 1.0 | `explicit` |
| 向量化失败或路由过程异常 | 默认路由 | `default_all` / `error` 等 |
| `kb_portraits` TopN 为空 | 无画像可用 | `no_portraits` |
| 智能路由：所有库 **原始分** max &lt; `ROUTING_ALL_LOW_THRESHOLD` | 列出最多 1000 个知识库，等价全库检索 | `low_confidence` |
| 智能路由：第一名与第二名 **归一化分** gap ≥ `ROUTING_GAP_DOMINANT` | 只选第一名 | `single_kb_dominant` |
| 否则且至少两库 | 取归一化分 Top2 | `dual_kb` |
| 仅一库有分 | 单库 | `single_kb` |
| 默认路由：能列出知识库 | 最多 100 个库均参与 | `default_all` |
| 无可用知识库 | 空列表 | `no_kb_available` |

## 4. 智能路由算法（当前实现）

### 4.1 常量（`router.py`）

| 常量 | 值 | 含义 |
|------|-----|------|
| `ROUTING_TOP_N` | 50 | 在 `kb_portraits` 全表做一次向量检索，取相似度最高的 N 个主题节点 |
| `ROUTING_TOP_K_PER_KB` | 5 | 每个 `kb_id` 在 TopN 命中的相似度列表中，**只保留最高的 K 个** 再聚合 |
| `ROUTING_DECAY_ALPHA` | 0.9 | 位置衰减底数：第 i 个（从 0 计数）权重为 α^i |
| `ROUTING_ALL_LOW_THRESHOLD` | 0.08 | 原始分全局最大值低于此阈值 → 认为匹配整体偏弱 → 全库 |
| `ROUTING_GAP_DOMINANT` | 0.3 | 归一化后第一名与第二名分差 ≥ 此值 → 判为单库主导 |

### 4.2 步骤 1：查询向量化

- 调用 `llm_manager.embed(texts=[query_text])`，与系统其他 embedding 一致。
- 失败则走 `_default_routing()`。

### 4.3 步骤 2：全局 TopN 检索

- `VectorStore.search_kb_portraits_topn(query_vector, limit=ROUTING_TOP_N)`（`vector_store.py`）。
- 实现为对集合 `kb_portraits` 的 **`query_points`**，返回列表项包含：`id`、`score`（相似度）、`kb_id`、`cluster_size`、`payload`。
- **说明**：`cluster_size` 仍会随结果返回，但 **当前 `_calculate_kb_scores_from_topn` 未使用**，仅作扩展或排查备用。

### 4.4 步骤 3：按知识库聚合原始分

对 TopN 结果按 `kb_id` 分组，收集每个库出现的所有 `score`（相似度）。

对每个 `kb_id`：

1. 将该库所有相似度 **降序排序**。
2. 取前 `ROUTING_TOP_K_PER_KB` 个（不足则全取），记为 `s_0, s_1, …`。
3. **加权平均**（权重随排位衰减，越靠前权重越大）：

\[
\text{Score}(KB) = \frac{\sum_{i=0}^{L-1} s_i \cdot \alpha^{i}}{\sum_{i=0}^{L-1} \alpha^{i}}, \quad L = \min(K, \text{该库在 TopN 中的节点数}),\ \alpha = 0.9
\]

设计意图简述：

- **Top-K 截断**：避免同一库在 TopN 中占大量席位时把分数「堆高」。
- **衰减加权**：最相关的几个主题节点主导该库得分，弱化长尾命中。

### 4.5 步骤 4：归一化

- `min-max` 到 `[0, 1]`：若所有值相同则各库记为 1.0。

### 4.6 步骤 5：决策

1. 若 `max(原始分) < ROUTING_ALL_LOW_THRESHOLD` → `low_confidence`，`target_kb_ids` = `list_knowledge_bases(limit=1000)` 的全部 id，置信度均 1.0。
2. 否则在 **归一化分** 上排序：
   - `gap = first_normed - second_normed`（仅一库时第二名为 0）
   - `gap >= ROUTING_GAP_DOMINANT` → 仅第一名，`single_kb_dominant`
   - 否则 → 取前两名（若存在），`dual_kb` 或 `single_kb`
3. `confidence_scores` 为选中库对应的 **归一化分**（非原始分）。

## 5. 返回结构 `RoutingResult`

| 字段 | 说明 |
|------|------|
| `target_kb_ids` | 选中的知识库 ID 列表 |
| `confidence_scores` | 各选中库的分数（显式/全库场景多为 1.0；智能路由为归一化分） |
| `routing_method` | 路由策略标签（见第 3 节表） |
| `total_candidates` | 参与打分的知识库数量（原始分字典大小） |
| `processing_time` | 秒 |
| `target_kbs` | 可选；每项含 `id`、`name`（来自 `KnowledgeBaseService.get_knowledge_base`）、`score`，供前端展示 |

填充 `target_kbs` 在 `route_query` 成功路径末尾通过 `_enrich_target_kbs` 异步完成。

## 6. 与检索链路的衔接

### 6.1 `resolve_to_qdrant_kb_ids`

- 前端或业务传入的 id 可能与 MinIO bucket 派生 id 一致，而 Qdrant 中文档与画像使用的是 **规范 kb_id**。
- `KnowledgeRouter.resolve_to_qdrant_kb_ids(kb_ids)` 对每个 id 调用 `KnowledgeBaseService._discover_kb_id_from_bucket_async`，解析成功则用规范 id，否则原样使用（去重、跳过空串）。
- `RetrievalService._perform_hybrid_search` 在调用 `HybridSearchEngine.search` **之前** 先做该解析，保证过滤条件命中向量库中的 `kb_id`。

### 6.2 显式指定知识库

- 请求上下文中 `kb_context["kb_ids"]` 非空时，**跳过** 画像检索与打分，直接以这些 id 为 `target_kb_ids`；混合检索前仍会 `resolve_to_qdrant_kb_ids`。

## 8. 日志与审计

- 原始按库得分：`_log_kb_scores_raw`（降序、便于对照 TopN 聚合结果）。
- 归一化得分与最终决策：`logger.info` 输出第一名、第二名、gap、阈值与 `routing_method`。
- `audit_log`：路由完成时记录查询摘要长度、目标库、置信度、耗时。
